System Features 

Frontend — React Dashboard
The operator-facing dashboard is built in React and consists of two rows of specialized panels:
•	Live Alert Queue — displays real-time, severity-sorted incident alerts with acknowledgment and escalation workflows.
•	Platform Heatmap — visualizes crowd density per zone with live updates to prevent overcrowding incidents.
•	Network Risk Map — a Leaflet-based geospatial map overlaying route-level risk scores across the rail network.
•	Camera Feed Panel — streams live RTSP camera feeds with AI-generated bounding box overlays for object/person detection.
•	Train Status Panel — shows real-time RTIS train positions, delays, and Kavach (collision avoidance) system status.
•	Grafana Panels — embeds time-series sensor trend dashboards for infrastructure health monitoring.
•	Incident Log — maintains a full audit trail of all events including response times for post-incident review.
•	Role-Based Views — tailors the dashboard UI for three access levels: Station staff, RPF (Railway Protection Force), and National Ops.
 
Backend Services
•	WebSocket Server — enables real-time data push to the dashboard using Socket.io, acting as a Kafka consumer bridge from sensor/camera streams.
•	 FastAPI REST Layer — handles authentication, historical data queries, and system configuration. 
•	 Alert Engine — runs Celery workers to evaluate incoming data against configurable severity rules and SLA thresholds. 
•	 Notification Dispatcher — delivers multi-channel alerts via FCM push notifications, Twilio SMS, and PA (Public Address) system API integration. 

 
 Data and ML Layer: 
•	Kafka— Five dedicated topics partition every data stream by domain; CCTV, GPS, IoT, signalling, and alerts, keeping sources fully decoupled so any component can be added or removed without touching the pipeline.
•	ML model Server — Four models run in parallel: YOLOv8 for threat detection, CSRNet for crowd density, MediaPipe for behaviour, and XGBoost+SHAP for explainable risk scoring,  all served via a single FastAPI inference endpoint.
•	Databases— .Four purpose-selected stores handle distinct workloads: PostgreSQL for incidents, Redis for real-time ML features and SLA timers, ClickHouse for high-speed time-series analytics, and Delta Lake for immutable long-term audit storage.
•	Observability —  Prometheus and Grafana monitor end-to-end alert latency SLAs, Kafka consumer lag, and model drift ensuring the system meets its sub-2-second detection-to-dashboard target at all times.


 




 











Integration

● HIGH RISK     ● MEDIUM RISK     → Fix / Recommendation

⚡ 1. Data Flow & Pipeline   ● HIGH
Camera & sensors → Kafka → WebSocket → React dashboard
Kafka broker failure kills Alert Queue, Heatmap & Camera Feed simultaneously
→ Add dead-letter queues + Socket.io auto-reconnect logic

🔗 2. API Contract   ● MEDIUM
FastAPI serves REST (auth, history, config) + WebSocket pushes live events
Schema drift between FastAPI & React causes silent data gaps in panels
→ Generate TypeScript types from OpenAPI spec; validate WebSocket payloads with Zod

🚨 3. Alert Lifecycle   ● HIGH
Sensor event → Kafka → Celery worker → severity check → SLA timer → dispatch
Celery crash mid-task silently drops the alert with no retry
→ Enable Celery retries with exponential backoff; persist alert state before dispatch

🕸 4. Cross-Component Dependencies   ● HIGH
WebSocket Server is a single point of failure for 4 live panels at once
Grafana & Incident Log are independent — safe if core backend fails
→ Add WebSocket redundancy with Redis adapter + sticky-session load balancing

🔐 5. Role-Based Access & Security   ● MEDIUM
3 roles: Station Staff · RPF Officers · National Ops
Role-only JWT tokens allow cross-zone API queries by station staff
→ Embed zoneId in JWT; enforce zone filtering server-side in FastAPI

🌐 6. Third-Party Integrations   ● MEDIUM
FCM push · Twilio SMS · PA system API · Grafana · Leaflet tiles
No unified retry/fallback across all three notification channels
→ Build a single dispatch queue with per-channel retry policies

📈 7. Scalability & Load   ● MEDIUM
WebSocket ~10K conn limit · Celery workers must match Kafka partitions
RTSP stream AI inference is CPU-heavy — shares resources with API layer
→ Offload GPU inference to dedicated worker pool; load test at 500 ops + 200 feeds
